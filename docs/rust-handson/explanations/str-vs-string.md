# 解説: &str と String

[01-basics.md](../01-basics.md) の「文字列: `&str` と `String`」セクションの補足。

## 一言で

- `String` = 文字列の本体（自分で持っている）
- `&str` = 文字列への見え方（誰かのを借りているだけ）

## メモリイメージ
ptrはポインタ
```
スタック                     ヒープ
┌──────────────────┐
│ s: String        │
│   ptr  ──────────┼─────→ [ h │ e │ l │ l │ o ]
│   len = 5        │       （ここを s が所有している）
│   cap = 5        │
└──────────────────┘

┌──────────────────┐
│ r: &str          │
│   ptr  ──────────┼─────→ （s のヒープの一部を指す）
│   len = 5        │
└──────────────────┘
```

`s: String` は「ヒープに `hello` というメモリを確保して、自分が持っている」状態。`r: &str` は「他人のヒープのこの位置を見ているよ」というポインタだけで、実体は持たない。

メモリの仕組みについては [スタックとヒープ](stack-and-heap.md) を参照。

## どこから来るか

```rust
let s1: &str = "hello";              // バイナリに焼き込まれた文字列への参照
let s2: String = String::from("hi"); // ヒープに新しく確保
let s3: &str = &s2;                  // s2 のヒープを指す参照
```

- リテラル `"hello"` はコンパイル時にバイナリに埋め込まれる → そこへの参照は `&str` で十分
- `String::from(...)` は動的にヒープを確保する → 持ち主が必要 → `String`
- `String` の前に `&` を付けると、その内容を指す `&str` が取れる

## 何が違うか

| 観点 | `String` | `&str` |
|---|---|---|
| メモリの持ち主 | 自分 | 他人 |
| ヒープ確保 | する | しない（参照だけ） |
| 中身を変更 | `mut` なら可（`push_str` など） | 不可 |
| 自分が消える時 | ヒープのメモリも解放 | 何もしない |
| サイズ | コンパイル時不明（実行時に伸縮） | 固定（ポインタ + 長さの 16 バイトのみ） |

## なぜ「持ち主」を意識する必要があるか

PHP / Go / Ruby は GC（ガベージコレクタ）が裏で「もう誰も使ってないメモリ」を勝手に回収してくれる。なので「文字列は誰のもの？」とか考えずに使える。

Rust は GC を持たない。代わりに「メモリの持ち主は同時に1人だけ。持ち主が消えたらメモリも解放」というルールで管理する。だから「これは持ち主（`String`）なのか、借りているだけ（`&str`）なのか」を常に区別する必要がある。

## 詰まる典型例

```rust
fn make_message() -> &str {
    let s = String::from("hello");
    &s   // ❌ コンパイルエラー
}
```

エラーになる理由:

- `s` は関数内ローカル変数
- 関数を抜けた瞬間、`s` の持っていたヒープメモリは解放される
- なのに `&s`（そのヒープへの参照）を返そうとしている
- 返した瞬間、参照先のメモリは無効 → ダングリングポインタ → Rust はこれをコンパイル時に弾く

正解は持ち主ごと返す:

```rust
fn make_message() -> String {
    String::from("hello")   // String を返す = 持ち主の権限ごと渡す
}
```

`&str` だけで済ませようとすると、こういう場面で詰まる。これが「`&str` だけ使うとすぐ詰まる」の意味。

## 雑な使い分けルール

最初はこれだけでOK。

| 場面 | 型 |
|---|---|
| 関数の引数（読むだけ） | `&str` |
| 関数の戻り値（新しく作って返す） | `String` |
| 構造体のフィールド | `String`（参照を持ちたいなら寿命指定が必要、最初は避ける） |
| リテラル | 自動で `&str` |

```rust
fn greet(name: &str) {                  // 引数は &str
    println!("Hello, {name}");
}

fn build_message(name: &str) -> String { // 戻り値は String（新規確保）
    format!("Hello, {name}!")
}

fn main() {
    let user: String = String::from("Yuhei");
    greet(&user);                       // String を渡す時は & で &str に
    greet("World");                     // リテラルはそのまま渡せる
    let msg = build_message(&user);
    println!("{msg}");
}
```

これだけで 9 割対応できる。

## 他言語との対応

| 言語 | 文字列の型 | 中身 |
|---|---|---|
| PHP / Python / Ruby | 1 種類 | GC 管理の参照型。所有を意識しない |
| Go | `string`（1 種類） | 不変参照。Rust の `&str` 寄りの動き |
| C | `char *` | 手動管理。所有が言語仕様にない |
| Rust | `String` + `&str` | 所有を型レベルで区別 |

特に Go の `string` は Rust の `&str` に近い「不変な参照」。Go では新しく作る時も同じ `string` 型で済むが、これは GC が裏で持ち主の管理をやってくれているから。Rust ではその役割を「`String` という型を明示的に使う」で表現している。

## まとめ

- `String` は持ち主（ヒープを所有）、`&str` は借り物（参照だけ）
- リテラルは自動で `&str`、動的に作るなら `String`
- 引数は `&str`、戻り値は `String`、を基本パターンにする
- 「持ち主は誰か」を意識する理由は、Rust に GC がないから

[02-ownership.md](../02-ownership.md) で「所有権」を本格的に扱う。今は「`String`/`&str` はメモリの持ち主の有無の違い」とイメージできれば十分。
