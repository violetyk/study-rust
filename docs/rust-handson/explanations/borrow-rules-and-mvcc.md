# 解説: 借用ルール vs SQL の MVCC

[02-ownership.md](../02-ownership.md) の借用ルール（不変参照と可変参照は同時に持てない）の補足。

> 同じ問題、別戦略。MVCC はランタイムでコストを払い、Rust はコンパイル時にチェックする

## 出発点の疑問

Rust の借用ルール:

```rust
let mut s = String::from("hi");
let r1 = &s;
let r2 = &mut s;    // ❌ 不変と可変は同時に持てない
println!("{r1}");
```

SQL なら、誰かが `BEGIN` してレコードを書き換えている最中でも、別プロセスからそのレコードを読める（MVCC 系の DB の場合）。なぜ Rust ではダメなのか?

## 結論: 解いている問題は同じ、戦略が違う

両者とも「読み書きの同時実行で起きる問題」を解決したい。でも解き方が違う。

| | SQL (MVCC) | Rust |
|---|---|---|
| アプローチ | 複数バージョンを持つ（書き手と読み手で別バージョンを見る） | 同時アクセス自体を禁じる |
| コストの所在 | ランタイム（MVCC の管理、古いバージョン保持、VACUUM） | コンパイル時（ボロチェッカー、実行時コストゼロ） |
| 安全性の担保 | トランザクション境界とロック・MVCC | 型システム |
| 失敗するタイミング | 実行時（デッドロック・直列化失敗） | コンパイル時 |

## なぜ Rust は「同時を禁じる」を選んだのか

理由は 2 つ。

### 1. ゼロコスト原則

Rust の哲学に「使わない機能のコストを払わない (zero-cost abstraction)」がある。MVCC のような仕組みはランタイムコスト（バージョン管理、メモリ余分、定期的な掃除）が必ず発生する。Rust は「コンパイル時にチェックすれば、ランタイムは何もしなくていい」を選んだ。

PostgreSQL の MVCC は確かに「書きと読みが並行できて素晴らしい」けど、その裏で:

- 古いバージョン (dead tuple) が溜まる → VACUUM が必要
- ヒープ膨張 → ディスク容量
- 長時間トランザクションが他のクエリをブロック (xmin horizon 問題)

このコストを「言語ランタイム」で常に払うのは、Rust が想定する用途（OS、組み込み、ゲーム、高速サーバ）に合わない。

### 2. メモリの「同じビット」を触る怖さ

DB の「レコード」は論理的な単位で、実体は別ファイル・別ページに複数バージョンを持てる。でも CPU メモリは違う。同じアドレスのビットは 1 種類しか存在しない。

```rust
let mut v = vec![1, 2, 3];
let r1 = &v[0];       // ヒープのこの位置を指す参照
v.push(4);            // ❌ もし許すと…
```

`push` の途中で `Vec` の容量が足りなくなると、ヒープに新しい領域を取って引っ越し (reallocation) する。引っ越し後、`r1` が指していたアドレスは「もう誰のものでもないメモリ」になる ―― これを許すと use after free（[02-ownership.md](../02-ownership.md) の演習 2-3 のテーマ）。

SQL なら「古いバージョンも残しておく」で解決できるけど、Rust ではメモリの古いバージョンを残すコストは現実的でない。だから「不変参照中は書き換え禁止」を強制する。

## 同じ問題への 3 つの戦略

言語・DB を横断して整理すると、読み書きの並行問題を解く戦略は大きく 3 つ。

| 戦略 | 採用例 | コスト | 安全性検出 |
|---|---|---|---|
| 同時アクセス禁止 (排他) | Rust の借用、ファイルロック、`Mutex` | 並行性が下がる | コンパイル時 or 実行時 |
| バージョニング (MVCC) | PostgreSQL, Oracle, Rust の `Arc<RwLock<T>>` | メモリ・管理コスト | 実行時 |
| 楽観的同時実行 (OCC) | DynamoDB の条件付き書き込み、ETag | 衝突時にリトライ | 実行時 |

Rust も MVCC 風の仕組みを使いたければ `Arc<RwLock<T>>` や `RefCell` を組み合わせれば実現できる。ただしそれは「明示的にランタイムコストを払う」ことになる ―― つまり Rust の借用ルールは「デフォルトはゼロコスト、必要なら明示的に払う」という設計。

## シングルスレッドでもこのルールが効く理由

ここが意外と見落とされがち。Rust の借用ルールは「データ競合（マルチスレッド問題）」だけのためではない。

```rust
let mut v = vec![1, 2, 3];
let r = &v[0];
v.push(4);            // ❌ シングルスレッドでも禁止
println!("{r}");
```

これがエラーになる理由はマルチスレッドではなく、メモリの reallocation で参照が無効になるから。シングルスレッドでもメモリ安全は壊れる。

つまり Rust の借用ルールは:

- マルチスレッドのデータ競合 (data race)
- シングルスレッドの iterator invalidation / use after free

の両方を、同じ仕組みで防いでいる。MVCC は「並行性」を解決する仕組みだが、Rust の借用ルールは「メモリ安全」と「並行性」を同時に解決している、と言ってもいい。

## Rust で MVCC 的なことをやりたい場合

どうしても「同時に読み書きしたい」なら、明示的にランタイムコストを払う。

```rust
use std::sync::{Arc, RwLock};

let data = Arc::new(RwLock::new(vec![1, 2, 3]));

// 読み手（複数 OK）
let r = data.read().unwrap();
println!("{:?}", *r);
drop(r);  // 明示的に読みロックを解放

// 書き手（排他）
let mut w = data.write().unwrap();
w.push(4);
```

`RwLock` は「読み手は何人でも、書き手は 1 人」を実行時に管理する（ロック取得のコスト発生）。これは Rust の借用ルールの「同じことをランタイムで動的にやる」版。

シングルスレッドで「借用ルールを実行時に判定したい」場合は `RefCell<T>` を使う。これも借用違反があったら panic（実行時エラー）になる。

```rust
use std::cell::RefCell;

let v = RefCell::new(vec![1, 2, 3]);
let r = v.borrow();           // 不変借用
let mut w = v.borrow_mut();   // ❌ panic: already borrowed
```

つまり Rust は「コンパイル時にチェックする借用」をデフォルトにしつつ、必要なら「実行時にチェックする借用」「マルチスレッドで使える借用」も用意している。コストを払う場所を選べる設計。

## MVCC との対比でわかること

| 観点 | MVCC (PostgreSQL) | Rust の借用 |
|---|---|---|
| 並行読み書きを許すか | 許す | 禁じる |
| 古いバージョンを保持するか | する（dead tuple） | しない |
| ロックの粒度 | 行 / ページ / テーブル | 値（変数）単位 |
| 「待たされる」場面 | 行ロック競合、コミット待ち | コンパイルエラーが先に出る |
| トレードオフ | ディスク・メモリ・管理コスト | 並行性が下がる |

DB は「同時実行性能が命」なので MVCC を選んだ。Rust は「ゼロコスト・メモリ安全が命」なので排他を選んだ。どっちが正しいではなく、用途に応じた選択。

## まとめ

- 借用ルールと MVCC は「読み書きの同時実行問題」への別アプローチ
- MVCC: ランタイムでバージョン管理、並行性を確保（管理コストあり）
- Rust: コンパイル時に排他を強制、ゼロランタイムコスト
- Rust が排他を選んだ理由は「ゼロコスト原則」と「メモリの古いバージョンを保持するコストの高さ」
- シングルスレッドでも借用ルールが効く理由は reallocation での use after free 防止
- Rust でも明示的に `Arc<RwLock<T>>` / `RefCell<T>` を使えばランタイムコストを払って並行アクセスができる
