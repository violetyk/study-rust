# 解説: ダングリング参照とライフタイム

[02-ownership.md](../02-ownership.md) の「ダングリング参照は作れない」の補足。

> 参照には「誰から借りたか」が型に書いてある。借用元が消える参照は型エラー

## ダングリングとは何か

「dangling = ぶら下がっている」。所有者がいなくなったメモリを指したままの参照のこと。

```
       (s が drop された後)
              │
              ↓
       ┌──────────────────────┐
       │ もう誰のものでもない  │  ← &s が指していた場所
       │ メモリ領域            │
       └──────────────────────┘
              ↑
              │ 参照だけ生き残ってる
        &s（呼び出し元）
```

C なら「アドレスはアドレスとして残るので、読み書きできてしまう」。でも書き換えられていたり、他の用途に再利用されていたりして、未定義動作 (undefined behavior) になる。これが [02-ownership.md](../02-ownership.md) 冒頭の三大バグの 1 つ、use after free。セキュリティ脆弱性の温床。

## Rust が「コンパイルが通らない」と言える理由

Rust は参照の生存期間（ライフタイム）を型に組み込んでいるから。

```rust
fn dangle() -> &String {     // ❌
    let s = String::from("hi");
    &s
}   // s は drop されるので、参照は無効
```

エラーメッセージ:

```
error[E0106]: missing lifetime specifier
 --> src/main.rs:1:16
  |
1 | fn dangle() -> &String {
  |                ^ expected named lifetime parameter
  |
  = help: this function's return type contains a borrowed value, but there is no value
    for it to be borrowed from
```

コンパイラはこう言ってる: 「戻り値に参照があるけど、その参照が『どの値から借りた』のか分からない」。

ライフタイム注釈を付ければ通る場合もあるけど、`dangle` のケースはどう書いても無理。`s` は関数内で作られて関数終了時に drop されるので、借用元が存在しなくなる。

## 修正パターン 3 つ

ダングリングを避ける典型的な書き方。

### パターン 1: 所有権を返す（move）

```rust
fn no_dangle() -> String {
    let s = String::from("hi");
    s                          // ← 所有権を呼び出し元に渡す
}
```

最も自然な解。参照を返したくなった時、まず「所有権を返せないか?」を考えるのが正解。Rust では「戻り値の move」はゼロコストで、所有権が呼び出し元にスッと移る。

### パターン 2: 引数の参照を経由する（ライフタイム引き継ぎ）

```rust
fn first_char(s: &String) -> &str {
    &s[0..1]                   // 引数 s のライフタイムを引き継ぐ
}
```

戻り値の参照は引数 `s` から借りたもの。引数が生きている間は戻り値も生きている、というのが Rust の暗黙ルール（lifetime elision rules）。

### パターン 3: `'static` を返す

```rust
fn greeting() -> &'static str {
    "hello"                    // 文字列リテラルは 'static
}
```

文字列リテラルはプログラム全体の寿命を持つので、参照を返しても安全。[static-str.md](static-str.md) を参照。

## 他言語との対比

| 言語 | ダングリング参照 | 検出のタイミング |
|---|---|---|
| C | 普通に作れて UB | 実行時クラッシュ or 静かに壊れる |
| C++ | 普通に作れて UB | コンパイラ警告（`-Wreturn-stack-address`）はあるが エラーにはならない |
| Go | escape analysis で自動的にヒープに移動 | コンパイラが自動対処 |
| Java / Ruby / PHP / Python | GC が参照を追うので発生しない | そもそも起きない |
| Rust | コンパイル時に拒否 | コンパイル時 |

### C の場合

```c
char* dangle() {
    char s[] = "hi";
    return s;          // ⚠️ コンパイル通る（警告は出る）
}
// 呼び出し側で s を使うと、スタックが上書きされて壊れたデータが返る
```

`s` はスタック上の配列。関数を抜けるとスタックフレームが巻き戻され、そのアドレスは他の関数呼び出しで再利用される。返ってきたポインタを使うと、運が良ければ昔のデータが残っていて気付かず、運が悪ければクラッシュ、最悪なのは「攻撃者に悪用できる状態」になる。

### Go の場合

```go
func noDangle() *string {
    s := "hi"
    return &s          // ✅ OK
}
```

Go コンパイラの escape analysis が「`s` の参照が関数外に逃げる」と判断し、`s` を自動でヒープに移動する。プログラマは気にしなくていいが、暗黙のヒープ確保コストが発生している。

### Rust の思想

Rust は「ヒープに置きたければ `Box`、`String` などを使う」と型で明示させる。だから `dangle()` を許さない代わりに、`no_dangle() -> String` のように「所有権を返す」と書けば素直に通る。

「コンパイラが裏で何かしてくれる」を避けて、「型を見ればコストと挙動が分かる」を優先する設計。

## ライフタイム注釈への伏線

`fn first_char(s: &String) -> &str` のような関数で、コンパイラが「戻り値の参照は引数 `s` から借りた」と推論できているのは、lifetime elision rules（ライフタイム省略規則）が働いているから。

複雑なケースではコンパイラが推論できなくなり、自分で `<'a>` のような注釈を書く必要がある。

```rust
fn longer<'a>(x: &'a str, y: &'a str) -> &'a str {
    if x.len() > y.len() { x } else { y }
}
```

`'a` は「`x` と `y` と戻り値が同じ寿命を共有する」を宣言している。`x` か `y` のどちらかが drop されたら戻り値も使えない、とコンパイラに伝える方法。

ダングリングが作れない理由は「ライフタイムが型システムに組み込まれているから」。具体的な仕組み（lifetime elision rules 3 ルール、明示的注釈、構造体に参照を持たせるパターン）は章 07 で深掘りする。

## なぜこの設計が安全と性能を両立できるか

ライフタイムはコンパイル時の概念で、コンパイル後のバイナリには何も残らない。

- C: ダングリング検出なし、性能は最高
- Go: escape analysis で自動ヒープ化（ランタイムコスト）
- Java / Ruby / PHP / Python: GC（大きなランタイムコスト）
- Rust: コンパイル時の型チェックのみ、ランタイムコストゼロ

「安全性と性能のトレードオフ」を回避するのが Rust の野心で、ライフタイムはその中核ピース。

## まとめ

- ダングリング参照 = 借用元が消えたのに参照だけ残る状態（use after free の元凶）
- Rust は参照の寿命を型に組み込むことで、コンパイル時に弾く
- 修正パターンは「所有権を返す」「引数のライフタイム引き継ぎ」「`'static`」の 3 つ
- C は野放し、Go は escape analysis で対処、Rust は型で明示
- ライフタイム注釈の本格的な話は章 07
