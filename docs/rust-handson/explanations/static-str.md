# 解説: &'static str と文字列リテラルの正体

[01-basics.md](../01-basics.md) の関数セクションに出てくる `&'static str` の解説。

> 文字列リテラルは「`static` キーワードの匿名版」と思うと一番分かりやすい。

## 一言で

`&'static str` = プログラムが終わるまで生き続ける文字列スライス。

```
&'static str
│ │      │
│ │      └── 文字列スライス（&str 本体）
│ └──────── ライフタイム名（'static は予約済みの特別なやつ）
└────────── 参照（&）
```

`'a` のような名前は「この参照はどれくらい有効か」をマークする。それの一番強いやつが `'static`。

## ライフタイムとは

参照（借り物）には「いつまで有効か」という寿命がある。借り物なので、貸し主が消えたら自分も無効になる。

```rust
let r;
{
    let s = String::from("hello");
    r = &s;  // r は s を借りる
}            // ← s が破棄される
println!("{r}"); // ❌ r は無効な参照（dangling reference）
```

これが借用チェッカに弾かれる典型例。「`r` のライフタイムは `s` のライフタイムより長くてはダメ」というルール。`'static` は「永遠に有効」を意味するので、こういう問題が起きない。

本格的な話は [07-lifetimes.md](../07-lifetimes.md) で扱う。

## なぜ文字列リテラルが 'static なのか

`"hello"` のような文字列リテラルは、コンパイル時にバイナリの読み取り専用領域に焼き込まれる。プログラム起動から終了まで、そのバイト列は同じ場所に居続ける = いつ参照しても有効 = `'static`。

```rust
let s: &'static str = "hello";
// "hello" はバイナリに埋め込まれている
// → プログラム終了まで居る → 'static
```

普段は `let s = "hello";` のように書いて型は推論に任せているが、正確な型はいつも `&'static str`。

## なぜ classify の戻り値が &'static str なのか

```rust
fn classify(n: i32) -> &'static str {
    if n < 0 { "negative" } else if n == 0 { "zero" } else { "positive" }
}
```

返すのは "negative" / "zero" / "positive" のいずれか。全部リテラル = 全部 `&'static str`。だから戻り値の型も `&'static str` と書く。

「`&str` だけじゃダメなの？」というと、ダメ。`-> &str` だとコンパイルエラーになる。理由は「この `&str` は誰のメモリを借りているの？」を Rust が判定できないから。

- 引数 `n: i32` は参照ではない → 引数から借りているわけでもない
- 関数内のローカル変数からは返せない（関数を抜けたら無効）
- 残る可能性は「永遠に生きているメモリから」 = `'static`

なので明示的に `'static` と書く必要がある。

## Rust の方針との繋がり

[function-return-type.md](function-return-type.md) で書いた「中身は推論、外側（シグネチャ）は明示」の話はライフタイムにもそのまま当てはまる。

```rust
fn classify(n: i32) -> &'static str {
//                     ────────────
//                     ↑ 関数の API として明示する
//                     「私が返す参照は永遠に有効ですよ」と契約

    if n < 0 { "negative" } else { ... }
//             ─────────
//             ↑ 推論される（"negative" は &'static str）
}
```

呼び出し側（外側）は型もライフタイムも見えていないと困る → 明示。中で何を選ぶか（推論で十分）→ 推論。

## 実は「ヒープではない」

「永遠に有効」と聞くと「ヒープに長く居続ける」と想像しがちだが、リテラルはヒープではなくバイナリの読み取り専用データ領域 (rodata) にある。

[stack-and-heap.md](stack-and-heap.md) のメモリ配置図を再掲:

```
高アドレス  ┌──────────────────┐
            │  スタック        │
            ├──────────────────┤
            │  ヒープ          │ ← 動的確保 (malloc/free)
            ├──────────────────┤
            │  BSS / Data      │ ← グローバル変数、static
            ├──────────────────┤
            │  Text / rodata   │ ← プログラムコード、リテラル ★ここ
低アドレス  └──────────────────┘
```

実行時の流れ:

```
1. ./ch01-basics を実行
2. OS が ELF/Mach-O ファイルを読み込み、各セクションを仮想メモリにマップ
   - .text   → コード領域（実行可能）
   - .rodata → 読み取り専用データ領域（リテラルがここ）
   - .data   → 書き込み可能なグローバル領域
3. main() の実行開始
4. classify() が呼ばれた時、すでにメモリ上にある "negative" のアドレスを返すだけ
```

「永遠に確保され続ける」というより、「バイナリそのものの一部としてメモリに常駐する」と言う方が正確。malloc / free の対象ではない、ヒープ管理コストもゼロ。

## バイナリの中身を覗いてみる

確認できる:

```fish
# バイナリ内の文字列を一覧
strings target/debug/ch01-basics | grep -E "^(negative|zero|positive)$"
```

実行ファイルの中に "negative" "zero" "positive" の文字列が直接含まれていることが見えるはず。`cargo build` した時点で、コンパイラがこれらの文字列をバイナリに埋め込んでいる。

## ヒープ確保との比較

```rust
let lit: &'static str = "hello";              // rodata（バイナリに焼き込み）
let owned: String = String::from("hello");    // ヒープに新たに確保
```

| 項目 | リテラル `&'static str` | `String::from("hello")` |
|---|---|---|
| 場所 | バイナリの rodata | ヒープ |
| 確保コスト | 0（プログラム起動時に既にある） | malloc 1 回 |
| 解放コスト | 0（プロセス終了時に OS が片付け） | Drop で free |
| 同じリテラルの重複 | コンパイラが重複排除（同じアドレスを返す） | 毎回別の確保 |
| 書き込み可能性 | 不可（read-only） | 可（`mut String` なら） |

リテラルがヒープ確保されないことには大きなメリットがある:

1. プログラム中に同じ文字列リテラルが何回出てきても、メモリ上は 1 コピーだけ
2. 関数を呼ぶたびに `String::from` するのと違って、確保コストが完全にゼロ
3. read-only なので並行アクセスしても安全（ロック不要）

```rust
fn classify(n: i32) -> &'static str {
    if n < 0 { "negative" } else if n == 0 { "zero" } else { "positive" }
}

// この関数を 1000 万回呼んでも、ヒープ確保は 1 回も発生しない
for i in -100..100 {
    let label = classify(i);   // rodata のアドレスを返すだけ
}
```

`String` 版を作ると毎回 malloc が走るので、性能差が出る。

## static キーワードとの関係（核心）

実は Rust には `static` キーワードという「明示的なグローバル変数」があって、リテラルはその匿名版とも言える。

```rust
// 名前付きグローバル定数
static GREETING: &str = "hello";

// 匿名（リテラル）
let s = "hello";
```

両方とも:

- 場所はバイナリの読み取り専用領域 (rodata)
- 寿命は `'static`
- 起動時にメモリ上に存在、プロセス終了で消える

文字列リテラルの正体は「コンパイラが裏で `static` を作って、そのアドレスを差し込んでいる」というイメージ。ほぼ匿名のグローバル変数。

## const と static の違い

ややこしいが、Rust には `const` と `static` の両方がある。

| | `const` | `static` |
|---|---|---|
| 場所 | 使われる場所にインライン展開 | 固定アドレスを持つ |
| アドレス | 取れない | `&FOO` でアドレスが取れる |
| 用途 | 定数（数値、短い文字列） | グローバル変数 |

```rust
const MAX: i32 = 100;             // 使われた箇所に 100 が埋め込まれる
static COUNT: i32 = 100;          // メモリ上の 1 箇所に存在、アドレスを取れる
```

普段の学習では `const` を使う場面が多い。`static` が必要になるのは、複数箇所から同じデータを参照したい時、`Mutex` などで共有可変状態を持ちたい時、など限られている。

## 他言語との対応

| 言語 | 同等の概念 |
|---|---|
| Rust | `static FOO: &str = "..."` / 文字列リテラル |
| C | `static const char* foo = "..."` / 文字列リテラル |
| Go | `const greeting = "..."`（コンパイル時定数） |
| Java | String pool（リテラルはインターン化されて共有） |
| PHP | リテラルは zend_string でインターン化（最適化次第） |

「同じ文字列リテラルが複数箇所に出てきたら、メモリ上は 1 コピーになる」というのも、Java の String pool や PHP の文字列インターン化と同じ発想。

## 補足: 'static でもヒープのこともある

ややこしいが、リテラル以外のルートで `'static` を作ると、ヒープにあるけど永遠に解放されないモノになる。

```rust
let s: String = String::from("hello");
let leaked: &'static str = Box::leak(s.into_boxed_str());
// ↑ これはヒープにある &'static str
// プロセス終了まで生き続ける（OS に返却されない = リーク）
```

特殊用途なので、普通の学習中には出てこない。

## まとめ

- `&'static str` = 永遠に生きる文字列スライス
- 文字列リテラルは自動でこの型を持つ
- リテラルは「`static` キーワードの匿名版」と思えば腹落ちしやすい
- 場所はヒープではなくバイナリの rodata（メモリ確保コストゼロ）
- 関数戻り値が「リテラルのどれか」なら `&'static str` と書く
- ライフタイムは「外側は明示」の Rust の方針の延長
- 本格的な話は [07-lifetimes.md](../07-lifetimes.md) で

今は「'static = 永遠に有効」「リテラルはこれ」「ヒープではなくバイナリに焼き込まれている」「`static` の匿名版」とだけ覚えておけば 01-basics の関数の例は理解できる。
