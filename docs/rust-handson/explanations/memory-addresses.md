# 解説: 高アドレスと低アドレス

[スタックとヒープ](stack-and-heap.md) の図に出てくる「高アドレス」「低アドレス」の話。

## メモリは巨大な番地付きバイト列

メモリは「先頭から1バイトずつ番号が振られた巨大な配列」と思うと分かりやすい。

```
番地 (アドレス)    内容
0x00000000      [ 1 バイト ]
0x00000001      [ 1 バイト ]
0x00000002      [ 1 バイト ]
   ...
0xFFFFFFFF      [ 1 バイト ]   ← 32bit システムなら最大ここまで
```

- 「低アドレス」= 番号が小さい方（`0x00000000` 側）
- 「高アドレス」= 番号が大きい方（`0xFFFFFFFF` 側）

64bit システムなら理論上の最大番地は `0xFFFFFFFFFFFFFFFF`。実際にプロセスが使えるのは仮想アドレス空間の一部に限られる。

## 実際の番地を見てみる

Rust で確認できる。

```rust
fn main() {
    let n: i32 = 42;
    let s: String = String::from("hello");

    println!("n のアドレス: {:p}", &n);
    println!("s のアドレス: {:p}", &s);
    println!("s の中身のアドレス: {:p}", s.as_ptr());
}
```

実行例:

```
n のアドレス: 0x16d0c2c4c
s のアドレス: 0x16d0c2c50
s の中身のアドレス: 0x130604070
```

`n` も `s` もスタック上にある変数なので、似たような大きい数字（`0x16d...`）の番地。`s.as_ptr()` はヒープを指すので、別の領域（`0x130...`）の番地になる。

## 図で「上が高アドレス」のお作法

メモリ配置図を描くとき、慣習で:

```
高アドレス  ┌──────────────┐
            │   スタック    │   番地が大きい方を上
            ├──────────────┤
            │              │
            │   ヒープ      │   番地が小さい方を下
低アドレス  └──────────────┘
```

としてスタックを上に、ヒープを下に描くことが多い。これは:

- スタックは「高アドレスから低アドレスに向かって伸びる」
- ヒープは「低アドレスから高アドレスに向かって伸びる」
- お互いに向かい合って成長する

という慣習を視覚化した図。お互いがぶつからないように、別の方向から伸ばしている。

## なぜこの方向か

歴史的な慣習が大きい。

- 古い CPU / OS では「スタックポインタを下げる = push、上げる = pop」が一般的（x86 もこの方式）
- スタック領域とヒープ領域を 1 つのメモリ空間で共存させるとき、両端から伸ばすと「真ん中のどこかでぶつかるまで」両方使える
- スタックオーバーフローは「スタックがヒープを侵食して衝突する」と起きる

```
高アドレス  ┌──────────────┐
            │   スタック    │ ← push されるたびに下に伸びる
            │      ↓        │
            │              │ ← この空白がぶつかったら「スタックオーバーフロー」
            │      ↑        │
            │   ヒープ      │ ← malloc されるたびに上に伸びる
低アドレス  └──────────────┘
```

実際の現代 OS では仮想メモリでもう少し複雑な配置になっているが、メンタルモデルとしてはこの「両端から伸びる」イメージで十分。

## まとめ

- 低アドレス = 番地の数字が小さい側
- 高アドレス = 番地の数字が大きい側
- メモリ配置図でスタックを上、ヒープを下に描くのは「スタックは下方向、ヒープは上方向に伸びる」慣習の表現
- 実用上は「スタックとヒープが別々の場所にあって、お互いに伸びていく」イメージで十分

番地そのものを意識する場面は多くないが、`{:p}` でポインタを出してみると「確かに別の場所にあるんだな」と実感できる。
