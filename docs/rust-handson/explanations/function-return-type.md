# 解説: 関数の戻り値の型はどこから来るか

[01-basics.md](../01-basics.md) の「関数」セクションの補足。

「式として値を返す」と「戻り値の型を `-> i32` で宣言する」という 2 つの話が混ざりやすいので整理する。

> 中身は推論、外側（シグネチャ）は明示、というのが Rust の方針。

## 結論: 「宣言」と「実体」の 2 箇所がある

```rust
fn add(x: i32, y: i32) -> i32 {
    x + y
}
//      ↑              ↑       ↑
//      引数の型        戻り値の  式の型
//      （宣言）       型の宣言    （実体）
```

`-> i32` は「この関数は i32 を返します」という宣言。`x + y` は「実際に返している式」で、これも結果が `i32`。

両者が一致していれば OK。一致していなければエラー。たとえば `x + y;`（セミコロン付き）にすると実体が `()` になり、宣言（`i32`）と食い違ってエラーになる。

## なぜ `x + y` が i32 なのか

型推論の連鎖になっている。

1. `x: i32` と `y: i32` が引数で宣言されている
2. `i32 + i32` の結果は `i32`（標準ライブラリの `Add` トレイトの仕様）
3. なので `x + y` 全体の型は `i32`

つまり「i32 を返すと宣言した」のではなく、「`x + y` を計算したら i32 になった」。式自体が型を持っている。

```rust
let z = x + y;  // z は i32 と推論される（型注釈なし）
```

Rust は式の型を引数の型から逆算してくれる。

## なぜ関数シグネチャでは型を省略できないのか

「式の型が分かるなら、関数の戻り値型も自動で分かるのでは？」と思うのが自然。Haskell や最近の Swift などは戻り値型を推論する。

Rust が関数シグネチャに型を必須にしているのは、こういう理由:

| 観点 | 解説 |
|---|---|
| 契約として明示 | 関数の API を読むだけで挙動が分かる。中身を読まなくていい |
| エラーを早期に出す | 戻り値の意図と実装がズレた時、シグネチャと比較して即エラーにできる |
| モジュール境界の安定 | 別 crate から使う関数の型が、実装の変更で勝手に変わらない |
| コンパイル速度 | 関数の中身を全部見て推論する必要がない |

「書く手間 < 読む人の認知負荷の軽減 + コンパイラの効率」という思想。

## 関数の中だけは推論されている

`let` の右辺など、関数の中身については Rust も型推論を効かせる。

```rust
fn add(x: i32, y: i32) -> i32 {
    let sum = x + y;        // sum: i32 と推論される
    let doubled = sum * 2;  // doubled: i32 と推論される
    doubled                 // 戻り値も i32
}
```

中身は推論、外側（シグネチャ）は明示、というのが Rust の方針。

## 整理

```rust
fn add(x: i32, y: i32) -> i32 {
//                       ───
//                       ↑ 「i32 を返します」と宣言（必須）
    x + y
//  ─────
//  ↑ x: i32, y: i32 から「結果は i32」と推論される
//  この型と上の宣言が一致しなければエラー
}
```

- 関数のシグネチャ `-> i32` は「人とコンパイラへの契約」
- 関数本体の `x + y` は「実体」
- 両者が一致するかをコンパイラがチェックする

つまり `-> i32` は「自動でこうなる」のではなく「私はこれを返すと言いました」と書いている、という違い。

## 補足: クロージャだと推論される

関数の中で定義する無名関数（クロージャ）だと型を省略できる。

```rust
let add = |x, y| x + y;  // 引数も戻り値も型推論
let z = add(1, 2);        // ここで初めて x, y, 戻り値が i32 に確定
```

「使われ方からの推論」がここでは効くので、Rust も一見推論できる風に見える。それでもシグネチャ付きの関数では推論しない、というのは「公開されるかもしれない関数の API を不安定にしないため」という Rust の哲学。
