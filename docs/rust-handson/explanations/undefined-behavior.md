# 解説: 未定義動作 (UB) と Rust の安全性保証

[02-ownership.md](../02-ownership.md) のダングリング参照（「未定義動作で死ぬ」）の補足。

> 普通のバグは結果が予測できる。UB は結果すら定義されていない ―― safe Rust はこれを型で根絶する

## UB とは何か

UB = Undefined Behavior（未定義動作）。C/C++ の言語仕様で「動作が定義されていない」とされている操作のこと。普通の「バグ」とは違って、もっと厄介な概念。

## 普通のバグと UB の違い

| | 普通のバグ | UB |
|---|---|---|
| 例 | 計算式を間違えた、条件分岐ミス | NULL ポインタ参照外し、ダングリング参照、配列範囲外 |
| 結果 | 「期待と違う値」が返る | 何が起きるか仕様上決まっていない |
| 再現性 | コードを見れば追える | コンパイラ・最適化・実行環境で挙動が変わる |
| デバッグ | 比較的しやすい | 悪夢 |

## なぜ UB が怖いか

「未定義」というと「ランダムに動く」と思われがちだが、本質はそうじゃなくて、コンパイラが UB を「絶対に起きないもの」として最適化に利用するから。

例: C で「ヌルポインタが NULL でなければ使う」みたいなチェックを書いても、コンパイラが「事前に `*p` してるからここまで来てるってことは p は NULL じゃないはず → このチェック消していい」と判断して、防御コードごと削除することがある。

結果として:

- デバッグビルドでは動く、リリースビルドで壊れる
- ある GCC バージョンでは動く、別バージョンで壊れる
- 似たコードが他の場所では動くから自分のロジックを疑い続ける

地獄。「動いてるからヨシ」が一番危ない世界。

## 典型的な UB（C/C++）

- ヌルポインタの参照外し: `*null_ptr`
- ダングリング参照の使用: `return &local_var;` の戻り値を使う
- 配列範囲外アクセス: `arr[100]` （要素 10 個しかないのに）
- 解放済みメモリへのアクセス (use after free): `free(p); *p;`
- 二重解放 (double free): `free(p); free(p);`
- 符号付き整数のオーバーフロー: `INT_MAX + 1`
- データ競合 (data race): 同期なしで複数スレッドが同じメモリに書き込む
- 初期化していない変数の読み出し

[02-ownership.md](../02-ownership.md) 冒頭の「Rust が解決したい三大バグ (double free / use after free / data race)」は、UB の典型例の中から特に「所有権・借用ルール」で防げるものを抜き出したもの、と言える。

## 他言語との対比

| 言語 | UB | 代わりに何が起きるか |
|---|---|---|
| C / C++ | あり（仕様で「動作未定義」と明記） | 何が起きても文句言えない |
| Go | ほぼなし（仕様で挙動が定義されている） | panic、ゼロ値、明確な振る舞い |
| Java | なし | `NullPointerException` などの例外 |
| Ruby / Python / PHP | なし | 例外、エラー |
| Rust (safe) | なし（保証されている） | コンパイル時に拒否、または panic |
| Rust (unsafe) | あり | C と同じ世界 |

Go や Java は「未初期化変数はゼロ値」「ヌル参照は例外」のように、起きうる悪い状況を「定義された挙動」に押し込めている。代わりにランタイムチェックのコストがある。

Rust は別アプローチで、safe Rust の範囲では「UB を引き起こすコード自体を書けない」という保証を型システムで提供する。逆に言うと、safe Rust の保証は「UB ゼロ」を約束している。

## safe Rust が保証していること

Rust 公式リファレンス ([Behavior considered undefined](https://doc.rust-lang.org/reference/behavior-considered-undefined.html)) に列挙されているもののうち主要なもの:

- データ競合
- 解放済みメモリへのアクセス (use after free)
- ヌルポインタや未整列ポインタの参照外し
- 初期化されていないメモリの読み出し
- ミュータブル参照のエイリアス（同時に複数の `&mut`）
- 不正な値（`bool` に 2 以上、`char` に無効な Unicode など）

safe Rust で書く限り、これらは絶対に起きない。これが Rust の言う「メモリ安全」「並行安全」の正体。

具体的に、所有権・借用ルールがどの UB を潰しているか:

| 仕組み | 潰している UB |
|---|---|
| 所有権 (1 つの所有者) | double free |
| 借用ルール (`&mut` は 1 本) | データ競合、iterator invalidation |
| ライフタイム | use after free、ダングリング参照 |
| 型システム (`Option<T>`) | NULL 参照外し |
| 配列・スライス境界チェック | 範囲外アクセス（実行時 panic で UB 化を防ぐ） |

つまり「メモリ安全 + 並行安全 = UB ゼロ」という設計目標があって、それを達成するために所有権・借用・ライフタイム・型システムを総動員している、というのが Rust の全体像。

## unsafe Rust の立場

`unsafe { ... }` ブロック内では、safe Rust が禁じている操作（生ポインタの参照外しなど）が書ける。そして UB が起きたら C 同様に何が起きるか分からない。

```rust
let p: *const i32 = std::ptr::null();
unsafe {
    let v = *p;  // ← UB: ヌルポインタ参照外し
}
```

`unsafe` は「ここから先は人間が責任を持ちます、コンパイラのチェックを部分的に外します」の宣言。標準ライブラリの内部実装や FFI（C との連携）で必要になるが、アプリケーションコードで書くことは稀。

重要なのは: unsafe ブロックの中でも「Rust の安全性保証」を破ってはいけないというのは変わらない。unsafe は「コンパイラに代わって人間が安全性を保証する」という契約。

## なぜ Rust が「メモリ安全と並行安全」を強調するのか

世界中のセキュリティ脆弱性の大半（Microsoft や Google の統計で 70% 前後と言われる）が、メモリ安全性の問題（特に use after free と out-of-bounds access）に起因している。これらは全部 UB。

Rust の野望は「C/C++ 並みの性能を保ちつつ、UB を構造的に根絶する」こと。GC を使わない（性能を犠牲にしない）、ランタイム検査を最小化する、それでもメモリ安全を達成する ―― そのために所有権という発明があった。

「メモリ安全」「並行安全」という言葉が抽象的に聞こえても、中身は「UB を起こすコードを書けないようにしてある」という具体的な保証です。

## まとめ

- UB (Undefined Behavior) = 仕様で動作が定義されていない操作
- 普通のバグと違って、コンパイラの最適化に利用されて予測不能な挙動になる
- C/C++ 由来の典型的な UB（double free / use after free / データ競合 / 範囲外 / NULL 参照外し など）が脆弱性の温床
- safe Rust は所有権・借用・ライフタイム・型システムを総動員して、これらの UB を構造的に根絶
- `unsafe` は安全性チェックを部分的に外す契約（標準ライブラリや FFI で使う）
- 「メモリ安全 + 並行安全 = UB ゼロ」が Rust の設計目標
