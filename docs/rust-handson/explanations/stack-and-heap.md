# 解説: スタックとヒープ

[01-basics.md](../01-basics.md) と [&str と String](str-vs-string.md) の補足。Rust の所有権を理解するための土台。

## メモリの全体像

1つのプロセスの仮想メモリは、上から下にこんな構造で並んでいる。

```
高アドレス  ┌──────────────────┐
            │  カーネル空間    │ ← OS が使う領域。ユーザープロセスから直接は見えない
            ├──────────────────┤
            │  スタック        │ ← 関数呼び出しごとにフレームを積む
            │  (stack)         │
            │      ↓           │
            │      ↓           │
            │                  │
            │      ↑           │
            │      ↑           │
            │  ヒープ          │ ← 動的に伸びるメモリ
            │  (heap)          │
            ├──────────────────┤
            │  BSS             │ ← 未初期化のグローバル変数
            │  (uninitialized) │
            ├──────────────────┤
            │  データ          │ ← 初期化済みのグローバル変数、static
            │  (initialized)   │
            ├──────────────────┤
            │  テキスト        │ ← プログラムコード、リテラル (rodata)
            │  (code)          │
低アドレス  └──────────────────┘
```

特徴:

- スタックとヒープは向かい合って成長する（お互いがぶつかると stack overflow）
- カーネル空間はユーザープロセスから直接読み書きできない（システムコール経由でのみアクセス）
- BSS / データ / テキストはバイナリの一部としてプログラム起動時にメモリにマップされる

「高アドレス」「低アドレス」については [メモリのアドレス](memory-addresses.md) を参照。

各セグメントの中身を整理:

| セグメント | 中身 | 例 |
|---|---|---|
| カーネル空間 | OS のコード・データ | （ユーザーは触れない） |
| スタック | ローカル変数、関数の戻り値、引数 | `let n = 42;` の `n` |
| ヒープ | 動的確保したメモリ | `String::from("hi")` の "hi" 本体 |
| BSS (Block Started by Symbol) | 未初期化のグローバル変数 | `static mut COUNT: u32;`（初期値なし） |
| データ | 初期化済みグローバル変数 | `static GREETING: &str = "hi";` の管理情報 |
| テキスト | プログラムコード、リテラル | `fn main() { ... }` のバイトコード、`"hello"` |

## 各プロセスは自分専用のメモリ空間を持つ

前述のメモリ図は「各プロセスから見た仮想アドレス空間 (virtual address space)」。実際の物理メモリ (RAM) とは別物で、OS とハードウェアの仕組みで仮想化されている。

```
プロセス A の仮想空間          プロセス B の仮想空間
┌──────────────┐              ┌──────────────┐
│ スタック     │              │ スタック     │
│ ヒープ       │              │ ヒープ       │
│ ...          │              │ ...          │
└──────────────┘              └──────────────┘
       │                              │
       ↓ MMU が変換                  ↓ MMU が変換
       │                              │
┌─────────────────────────────────────────────┐
│  物理メモリ (RAM)                            │
│  [いろんなプロセスのデータが混在して配置]     │
└─────────────────────────────────────────────┘
```

特徴:

- 各プロセスは「自分だけの広大なメモリ空間」を持っているように見える
- 同じ仮想アドレス（例: `0x1000`）でも、プロセス A と B では別の物理メモリを指している
- プロセス間でメモリが分離される（一方が他方のメモリを直接読み書きできない）

### MMU の役割

MMU (Memory Management Unit) は、CPU 内部のハードウェアで、論理アドレス（仮想アドレス）を物理アドレスに変換する。OS が用意した「ページテーブル」を参照しながら高速に変換する。

### メリット

仮想化のおかげで成立しているもの:

- セキュリティ: あるプロセスが他のプロセスのメモリを覗けない（仮想空間が分離されている）
- 安定性: 1 つのプロセスがクラッシュしても他に波及しない
- 利便性: プロセスは「連続した広大な空間」を仮定してコードを書ける（実際は飛び飛びの物理メモリでも、MMU が連続しているように見せる）
- ページング・スワップ: 物理メモリが足りなくなったら、使われていないページをディスクに退避して、別のプロセスに物理メモリを譲ることができる

### Rust 学習との関係

`{:p}` で表示されるアドレス（[memory-addresses.md](memory-addresses.md)）は仮想アドレス。同じ Rust プログラムを 2 回起動しても、表示されるアドレスは似たような値になる（プロセスごとに同じ仮想空間レイアウトを持つから）。

メモリの「持ち主」「Drop」「所有権」の話は、すべて 1 つのプロセス内の仮想アドレス空間の話。MMU と物理メモリの仕組みは普段意識する必要はないが、「なぜプロセスごとに独立してメモリ管理できるか」の前提として知っておくと安心感がある。

OS がプロセスをどう管理するか (PCB、スケジューラ、コンテキストスイッチ、IPC、スレッド) については [プロセスとスレッド (OS の基礎)](process-and-thread.md) にまとめた。

## スタックの特徴

LIFO（後入れ先出し）の高速メモリ領域。関数呼び出しと運命を共にする。

```rust
fn outer() {
    let a = 10;        // outer のスタックフレームに a を積む
    inner();
    // outer 終了 → a も解放
}

fn inner() {
    let b = 20;        // inner のフレームに b を積む
    // inner 終了 → b は解放、フレームごと巻き戻る
}
```

スタックの動きを時系列で:

```
時点1: outer 開始    時点2: inner 呼び出し    時点3: inner 終了
┌──────────┐         ┌──────────┐             ┌──────────┐
│          │         │          │             │          │
│          │         │ b = 20   │             │          │
│ a = 10   │         │ a = 10   │             │ a = 10   │
│ outer    │         │ inner    │             │ outer    │
│ frame    │         │ frame    │             │ frame    │
└──────────┘         │ outer    │             └──────────┘
                     │ frame    │
                     └──────────┘
```

| 観点 | スタック |
|---|---|
| 速度 | 圧倒的に速い（ポインタ移動だけ） |
| サイズ | 小さい（OS デフォルト 1〜8MB） |
| 寿命 | 関数のスコープと一致 |
| サイズ要件 | コンパイル時に固定サイズ必須 |
| 解放 | 自動（関数を抜けた瞬間） |

## ヒープの特徴

動的に確保・解放する大きな共有メモリ領域。スコープに縛られない。

```rust
fn main() {
    let s = String::from("hello");
    // ヒープに "hello" を確保し、ポインタはスタックの s へ

    other_fn(&s);
    // s の参照を渡す。ヒープのメモリはまだ s が所有

    // main 終了時に s が破棄され、ヒープも解放
}
```

| 観点 | ヒープ |
|---|---|
| 速度 | スタックより遅い（OS にメモリ要求、管理コスト） |
| サイズ | 大きい（仮想メモリ範囲、実質 GB 級） |
| 寿命 | 自分で管理（Rust なら所有権、他言語なら GC や手動） |
| サイズ要件 | 実行時に決まってOK（伸縮可能） |
| 解放 | 言語依存（Rust = 所有者の Drop、Go/Java = GC、C = `free`） |

## なぜこの 2 つに分かれているか

両方の良さを使い分けるため:

- スタック = 速い、自動管理、でも小さくて固定サイズ必須
- ヒープ = 遅い、管理が必要、でも大きくて動的サイズOK

「サイズが分かっていてすぐ捨てるもの」はスタックで超高速処理、「大きい・動的・長生き」はヒープで柔軟性、と棲み分け。

## Rust 型の振り分け

Rust の型は基本「スタック行き」と「ヒープ行き」が型レベルで決まっている。

| 型 | 配置 | 理由 |
|---|---|---|
| `i32`, `u64`, `f64`, `bool`, `char` | スタック | サイズ固定 |
| `(i32, f64)` のタプル | スタック | 中身のサイズが固定 |
| `[i32; 5]` 固定長配列 | スタック | サイズ固定 |
| `&str` | スタック（中身は他所） | ポインタ + 長さだけ持つ |
| `&T` 参照全般 | スタック | 同上 |
| `String` | ヒープ（管理情報はスタック） | 動的サイズ |
| `Vec<T>` | ヒープ（同上） | 動的サイズ |
| `Box<T>` | ヒープ（同上） | 「強制的にヒープに置く」型 |
| `Rc<T>`, `Arc<T>` | ヒープ（同上） | 共有所有 |

`String` や `Vec` は「実体はヒープにあるけど、その実体への参照（ポインタ・長さ・容量）はスタックに置かれる」という二段構成になっている。

### 因果の方向: 「ローカル変数だから」ではなく「型が固定サイズだから」

「`i32` や `bool` はローカル変数として使うことが多いからスタックに置かれる」は現象としては正しい。でも因果は逆で、決めるのは「型の性質」と「作られる場所」の 2 段階。

| 型の性質 | その変数の作られ方 | 結果 |
|---|---|---|
| サイズ固定 (Copy) | ローカル変数として宣言 | スタック |
| サイズ固定 | `Box` / `Vec` / 構造体経由 | ヒープ |
| 動的サイズ | ローカル変数として宣言 | 管理情報のみスタック、本体ヒープ |

具体例:

```rust
let n: i32 = 42;                    // i32 + ローカル → スタック
let b: Box<i32> = Box::new(42);     // i32 + Box → 値はヒープ、b 自身（ポインタ）がスタック
let v: Vec<i32> = vec![1, 2, 3];    // i32 が 3 個 → ヒープ。管理情報 (ptr/len/cap) がスタック
let s: String = String::from("hi"); // String もローカル変数だけど、本体はヒープ
```

「ローカル変数 = スタック」と短絡すると、`Box<i32>`（ローカル変数なのに値はヒープ）や `String`（ローカル変数なのに本体はヒープ）を見たときに混乱する。根っこのルールは「型のサイズが固定 + そのまま値として配置される」→ スタック。

Go の escape analysis との対比で言うと、Go は「変数の参照が関数外に逃げるか」をコンパイラが解析して必要ならヒープに移すので、同じコードでもコンパイラの判断でスタック/ヒープが変わる。Rust は型を見れば配置が分かる ―― この違いも「Rust では型が配置を決める」の現れ。

## 具体例で見る

```rust
fn main() {
    let n: i32 = 42;
    let arr: [i32; 3] = [1, 2, 3];
    let s: String = String::from("hello");
    let v: Vec<i32> = vec![10, 20, 30];
}
```

メモリの様子:

```
スタック (main のフレーム)              ヒープ
┌───────────────────────┐
│ n: i32 = 42           │
│                       │
│ arr: [1, 2, 3]        │   ← 配列も中身ごとスタック
│                       │
│ s: String             │       ┌────────────────┐
│   ptr  ───────────────┼─────→ │ h e l l o      │
│   len = 5             │       └────────────────┘
│   cap = 5             │
│                       │
│ v: Vec<i32>           │       ┌────────────────┐
│   ptr  ───────────────┼─────→ │ 10 │ 20 │ 30   │
│   len = 3             │       └────────────────┘
│   cap = 3             │
└───────────────────────┘
```

`s` も `v` も「スタックに小さな管理情報、ヒープに本体」というレイアウト。

## 関数を抜ける時に何が起きるか

```rust
fn main() {
    let s = String::from("hello");
    // ... 何か処理 ...
}   // ← ここで何が起きる？
```

main を抜ける時:

1. スタックの `s`（管理情報）が解放される
2. その時、Rust は `Drop` トレイトの仕組みで「`s` が持っていたヒープメモリも解放しろ」を自動で実行
3. ヒープの `hello` のメモリが OS に返却される

これを Rust は「所有権が消える時に Drop が走る」と表現する。GC のない世界で、決定論的にヒープメモリを管理する仕組み。

## 他言語との比較

| 言語 | スタック/ヒープの意識 | ヒープ解放の方法 |
|---|---|---|
| C | 必要 | 手動 (`free`) |
| C++ | 必要 | 手動 or RAII |
| Rust | 必要（型で表現） | 自動（所有権 + Drop） |
| Go | 隠蔽（escape analysis が決める） | GC |
| Java | 隠蔽（プリミティブ以外は全部ヒープ） | GC |
| PHP / Python / Ruby | ほぼ意識しない（全部ヒープ） | GC |

Go は escape analysis という最適化で「この変数はスタックに置けるか？」を自動判定している。同じコードでもコンパイル次第でスタック/ヒープが変わる。Rust はそれを型でコントロールする思想。

PHP / Python / Ruby は「全部ヒープにオブジェクト、参照を渡すだけ、GC が勝手に回収」なので、そもそもこの区別が見えない。

## なぜ Rust では特に意識するか

GC がないから。

- GC ありの言語: 「いつ解放するか」を機械が決めてくれる → 所有を意識する必要なし
- Rust: 「持ち主が誰か」を型で追って、持ち主が消える瞬間に解放 → スタックとヒープと所有権がセットで意味を持つ

その代わり Rust は:

- ランタイム GC がない（性能的に有利）
- メモリリークやダングリングポインタをコンパイル時に検出（安全性）

このトレードオフを成立させているのが、所有権と借用のシステム。そしてそれを理解するには「どこに何のデータがあるか」（スタック/ヒープ）の感覚が必須。

## まとめ

- スタック = 速い・固定サイズ・関数スコープと連動
- ヒープ = 遅い・動的サイズ・寿命を自分で管理
- Rust は型レベルでこの 2 つを分け、所有権で寿命を追う
- `String` や `Vec` は「スタックに管理情報、ヒープに本体」の二段構成
- GC がないからこそ、この区別が意味を持つ

[&str と String](str-vs-string.md) の話に戻すと:

- `&str` = スタックに「他所のヒープを指すポインタ + 長さ」
- `String` = スタックに「自分が持っているヒープへのポインタ + 長さ + 容量」

「所有者かどうか」 = 「ヒープのこの領域を解放する責任があるかどうか」。これが [02-ownership.md](../02-ownership.md) の所有権の話に直接つながる。
