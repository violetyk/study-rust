# 解説: スライスとは

[01-basics.md](../01-basics.md) に出てくる「文字列スライス」など、Rust の「スライス」の正体。

> 「切った物」の語感そのままで、メモリ上の一区画を指す参照。

実データを持つわけではなく、「ここからここまで見てるよ」というラベルが本質。

## 図解

```
元データ:  [ 1 │ 2 │ 3 │ 4 │ 5 │ 6 │ 7 ]
                ↑           ↑
                └─切り取り範囲─┘

スライス: → ここを指す参照（ポインタ + 長さ）
```

「元の塊から切り出した一区画への参照」。中身を持ち歩くわけではない。

## 文字列で見る

```rust
let s = String::from("hello world");
let hello: &str = &s[0..5];    // "hello"  0 文字目から 5 文字目未満
let world: &str = &s[6..11];   // "world"  6 文字目から 11 文字目未満
let all: &str = &s[..];        // 全体（範囲省略）
```

```
s: String
   ptr ───→ ┌───────────────────────┐
            │ h e l l o   w o r l d │
            └───────────────────────┘
              ↑   ↑     ↑           ↑
            [0..5]      [6..11]
            hello       world
```

`hello` は新しい文字列を作っているわけではない。`s` のヒープのこの位置からこの長さだけ見ている、という参照。

## 配列・Vec も同じ

```rust
let arr = [10, 20, 30, 40, 50];
let middle: &[i32] = &arr[1..4];   // [20, 30, 40]

let v = vec![10, 20, 30, 40, 50];
let tail: &[i32] = &v[2..];         // [30, 40, 50]
```

`&[T]` が「`T` の配列のスライス」、`&str` が「文字列（UTF-8 バイト列）のスライス」。命名が統一されている。

## なぜ「スライス」と呼ぶか

英語の "slice" は元々「薄切り、一切れ」の意味で、これがそのまま日本語の「スライスチーズ」になっている。プログラミング用語としても同じ語感:

| 元 | スライス |
|---|---|
| 食パン 1 斤 | 1 枚切り出したやつ |
| `String` / `Vec` / 配列 | 一部分への参照 |
| 「全体を持つ人」 | 「一部だけ見たい人」 |

切り出したスライスは「元の食パンの一部であって、独立した食パンではない」。Rust のスライスも「元の `String` / `Vec` の一部であって、独立した文字列・配列ではない」。だから所有権を持たない。元が消えたらスライスも無効になる。

## 文字列リテラルもスライス

`&'static str`（[static-str.md](static-str.md)）の話に戻ると、リテラル `"hello"` も実はスライス。

```rust
let s: &'static str = "hello";
```

これは「バイナリの rodata に焼かれた `hello` という文字列の、全範囲を見るスライス」。元データは rodata、その全部または一部を見ている参照、という構造。

```
バイナリの rodata セクション:
[..."hello"...他のリテラル...]
       ↑ここを指す
       &'static str
```

リテラルもスライス、`String::from(...)` から `&s[0..5]` で取り出すのもスライス、扱いとしては同じ `&str`。

## スライスの正体: ポインタ + 長さ

`&[T]` や `&str` の中身はこういう構造:

```
&[T] / &str (16 バイト = 64bit システム上)
┌──────────────────────────────────┐
│ ptr: データの先頭アドレス        │
├──────────────────────────────────┤
│ len: 何要素分（または何バイト）  │
└──────────────────────────────────┘
```

これだけで「どこから・どこまで」が決まる。元データへの参照と長さ、の 2 つの情報。「fat pointer」（太いポインタ）と呼ばれることもある。

## 比較: スライスとコレクション

| | スライス（`&[T]` / `&str`） | コレクション（`Vec<T>` / `String`） |
|---|---|---|
| 持ち主 | 借り物（参照） | 自分 |
| 確保 | しない | ヒープに確保 |
| サイズ | ポインタ + 長さ（16 バイト） | ポインタ + 長さ + 容量（24 バイト） |
| 中身を変更 | 不可 | 可（mut なら） |
| 元データが消える時 | 無効になる | 自分も消える（ヒープ解放） |

## 他言語との対応

| 言語 | 似た概念 |
|---|---|
| Rust | `&[T]` / `&str` |
| Go | `[]T` slice（所有権を持つ可変長コンテナ寄り） |
| Python | `list[1:4]` で部分リスト（コピーを作る） |
| PHP / Ruby | `array_slice` / `slice` メソッド（コピーを作る） |
| C++ | `std::span<T>` / `std::string_view`（C++17 以降） |

注意点として:

- Python / PHP / Ruby の「スライス」はだいたい新しいコピーを作る
- Rust のスライスは「参照だけ」なのでコピーしない（コスト低い）
- 一番近いのは C++ の `std::string_view` / `std::span`

Go の slice はちょっと違って、Vec / String 寄りの「所有権を持つ動的配列」になっている。同じ「slice」という言葉でも、Rust の方が「単なる参照」寄り。

## まとめ

- スライス = 配列・文字列の一部分を指す参照
- 「切ったやつ」のイメージで正解（食パンから 1 枚切り出す感じ）
- 中身は「ポインタ + 長さ」だけ、元データを所有しない
- `&[T]`（型 T のスライス）と `&str`（文字列スライス）が代表
- `String` `Vec` `配列` から `&data[a..b]` で取り出せる
- 文字列リテラルもスライス（rodata 上のデータへの参照）

「切った物」の語感そのままで、メモリ上の一区画を指す参照、と思っておけば OK。
