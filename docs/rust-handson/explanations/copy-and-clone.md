# 解説: Copy と Clone

[02-ownership.md](../02-ownership.md) の Copy トレイトの補足。

> Copy は暗黙、Clone は明示。コピーのコストが見えるように分けてある

## まず用語

- トレイト (trait) = 他言語のインターフェースに近い概念。型に「この振る舞いができますよ」を宣言する仕組み
- `Copy` = 「この型は値ごと（ビットごと）コピーしてよい」という印
- `Clone` = 「コピーするときの手順をちゃんと定義してある」型

`Copy` を実装している型は move ではなく自動でコピーされる。

## なぜ「move が基本」で「Copy は例外」なのか

設計の順序が大事。

```rust
let x = 5;
let y = x;
println!("{x} {y}");   // ← もし move が基本なら、これがエラーになる
```

毎回 `i32` でこれが書けないと、Rust が現実的に書けない言語になってしまう。とはいえ「すべての型をコピー可能にする」と `String`/`Vec` で double free が起きる（[所有のラインは 1 本だけ](move-and-ownership-line.md) を参照）。

そこで Rust の妥協:

- 基本動作: move（所有権の移動）
- 例外: 「コピーしても困らない型」だけ自動コピー（= `Copy` トレイト）

「コピーしても困らない」の判定基準:

1. ヒープを持っていない（コピーしても double free のリスクがない）
2. スタック上の単純なビット列で完結する（memcpy 1 発で複製できる）

この 2 つを満たす型だけが `Copy` を実装できる。

## Copy 可能な型・不可能な型

```rust
// Copy 可能（スタックで完結）
let a: i32 = 1;
let b = a;           // コピー。a もまだ使える
println!("{a} {b}");

// Copy 不可（ヒープを持つ）
let s = String::from("hi");
let t = s;           // move。s はもう使えない
// println!("{s}");  // ❌

// 構造体で Copy になれるのは「全フィールドが Copy」のときだけ
#[derive(Copy, Clone)]
struct Point { x: i32, y: i32 }

// これは Copy にできない（String を含む）
// #[derive(Copy, Clone)]  ← コンパイルエラー
struct Person { name: String, age: u32 }
```

主な Copy 実装型:

- すべての整数・浮動小数（`i32`, `u64`, `f64` など）
- `bool`, `char`
- すべて Copy な要素のタプル
- 固定長配列（要素が Copy なら）
- 参照 `&T`（参照そのものはアドレス値なので Copy）

## Copy と Clone の関係

`Copy` と `Clone` は別物だけど親戚。

| | Copy | Clone |
|---|---|---|
| 発動 | 暗黙（`let y = x;` で自動） | 明示（`x.clone()` を書く必要あり） |
| 内容 | 単純な memcpy（ビットコピー） | 任意の処理を書ける（ヒープも複製可） |
| コスト | 安い前提 | 高くなり得る（だから明示的に書かせる） |
| 適用範囲 | スタック完結型のみ | どんな型でも実装可能 |

そして:

- `Copy` を実装する型は必ず `Clone` も実装しないといけない（言語ルール）
- `Clone` だけ実装している型は普通にある（例: `String`, `Vec`）

```rust
// String は Clone はあるが Copy はない
let s1 = String::from("hi");
let s2 = s1.clone();   // ✅ 明示的に clone（ヒープも複製、コスト大）
println!("{s1} {s2}");
```

なぜ「`Copy` なら `Clone` も必須」かというと、`Clone` は「コピー方法を明示」のインターフェース、`Copy` は「その上で暗黙コピーを許す」というマーカー。順序として `Clone` が下位、`Copy` が上位の関係。

```
        Clone (どんなコピーロジックでも書ける)
          ↑
          │ Copy を実装するなら Clone も必須
          │
        Copy (memcpy で済む型に付けるマーカー)
```

## なぜこの設計が良いか（Rust の哲学）

「コピーは原則として明示的に書く」というのが Rust の思想。

- ヒープを持つデータの複製は高い操作 → 明示的に `.clone()` を書かせて意図を可視化
- スタックで完結する小さい値は暗黙コピー OK → 書き味を犠牲にしない

他言語と比べると性格がハッキリする:

| 言語 | 値のコピー挙動 | コストの可視性 |
|---|---|---|
| PHP | 配列・オブジェクトは参照、スカラーは値コピー | 言語仕様を覚えるしかない |
| Python / Ruby | すべて参照渡し、`deepcopy` で深いコピー | 浅い・深いを意識しないとハマる |
| Go | 値型はコピー、スライス/マップは参照コピー | 型ごとに挙動が分かれる |
| C++ | デフォルトはコピー（コピーコンストラクタ）、`std::move` で明示 move | コピー演算子の定義次第 |
| Rust | デフォルトは move、`Copy` 付きの型だけ暗黙コピー | 型を見れば一目で分かる |

Rust は「Copy が付いてるかどうか」が型に書いてあるので、コードを読めばコストが分かる。「`Vec<MyStruct>` を渡すと clone が必要だな（高い操作）」「`u64` を渡すならコピーで OK（安い）」が一発で判断できる。

## まとめ

- move が基本、Copy は例外（コピーしても安全な型だけ暗黙コピーを許す）
- 判定基準は「ヒープを持たない」「memcpy で済む」
- `Copy` は暗黙、`Clone` は明示。`Copy` を付けるには `Clone` も必須
- 「コピーは明示が原則」が Rust の思想 ―― コストを型で可視化する
