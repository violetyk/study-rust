# 解説: ジェネリクスとターボフィッシュ

[02-ownership.md](../02-ownership.md) の演習 2-2 で `Iterator::sum` のシグネチャを読み解く時の補足。本格的な深掘りは章 05 で。

> `<S>` で型を抽象化、`::<>` で型を明示。1 つの関数で複数の型に対応できる

## 出会ったシグネチャ

```rust
pub fn sum<S>(self) -> S
where
    Self: Sized,
    S: Sum<Self::Item>,
```

`<S>` って何? `where` って何? という疑問への入口。

## `<S>` は「呼び出し側が決める型」

`<S>` は関数のシグネチャで「これから S という型変数を使います」と宣言しているもの。中身が何かはコンパイル時に決まる「型のパラメータ」。

```rust
pub fn sum<S>(self) -> S
//      ^^^      ^
//      宣言    戻り値の型 = 呼び出し側が決めた型
```

`S` は「ジェネリック型パラメータ (generic type parameter)」と呼ばれる。型を抽象化して関数を書ける機能。

## なぜジェネリクスを使うのか

`Vec<i32>` から合計を取る時、戻り値を `i32` にしたい人もいれば、オーバーフロー防止に `i64` で受け取りたい人もいる。型ごとに `sum_i32()`, `sum_i64()`, `sum_f64()` と関数を分けるのは無駄。

ジェネリクスを使えば 1 つの関数で全部対応:

```rust
let v = vec![1, 2, 3];

let sum: i32 = v.iter().sum();    // S = i32
let sum: i64 = v.iter().sum();    // S = i64
let sum: f64 = vec![1.0, 2.0].iter().sum();  // S = f64
```

## どうやって S が決まるか

2 通り。

### 1. 型推論（コンテキストから）

```rust
fn total_and_max(v: &[i32]) -> (i32, i32) {
    (v.iter().sum(), *v.iter().max().unwrap())
    //        ^^^^^
    //        戻り値が (i32, i32) なので、ここの sum も i32 と推論される
}
```

「戻り値の型」から逆算して、`.sum()` の S が `i32` と決まっている。型推論が後ろから引っ張られる例。

```rust
let sum: i32 = v.iter().sum();
//      ^^^   ← 左辺の型注釈から S = i32 と推論
```

### 2. ターボフィッシュ `::<>` で明示

```rust
let sum = v.iter().sum::<i32>();
//                    ^^^^^^^
//                    これがターボフィッシュ。S = i32 と直接指定
```

`::<>` は Rust の有名な記法で「ターボフィッシュ (turbofish)」と呼ばれる。形が魚っぽいから。

なぜ `::<>` と書くか:

- 普通の `<>` だと比較演算子 `<` と曖昧になる
- `name<T>(...)` を式中で書くと「name 未満で開始?」とパーサが混乱する
- だから「型パラメータの開始は `::<` ですよ」と明示するためにコロンを 2 つ足した

```rust
// あいまい (パース不能)
let x = foo<i32>();    // ❌ foo < i32 > () と読まれる

// 明確 (turbofish)
let x = foo::<i32>();  // ✅ foo の型パラメータは i32
```

ターボフィッシュが必要になる場面:

- `.collect::<Vec<i32>>()` で集めたい型を指定
- `.parse::<i32>()` でパース結果の型を指定
- `Vec::<i32>::new()` で空 Vec の型を指定（`Vec::new()` 単体だと推論できない場面で）

## `where` の制約は何のため

```rust
where
    Self: Sized,
    S: Sum<Self::Item>,
```

- `Self: Sized` ―― イテレータ自身のサイズが決まっている必要がある（学習中はスルーOK）
- `S: Sum<Self::Item>` ―― 「S は『要素 (Item)』を集計して S を作れる型」でないとダメ、という制約

これを「トレイト境界 (trait bound)」と呼ぶ。「S は何でもいい」ではなく「S は Sum トレイトを実装している型に限る」という縛りを付けている。

`Sum` トレイトを実装している型なら何でも入れられる、というのを型レベルで保証している:

- `i32` は `Sum<i32>` も `Sum<&i32>` も実装 → `v.into_iter().sum::<i32>()` も `v.iter().sum::<i32>()` も OK
- `String` は `Sum<String>` を実装 → 文字列を連結できる
- 自分で `Sum` トレイトを実装すれば、独自型でも `.sum()` が使える

`where` 句を書かないと、`S` に何が入ってもよくなって、`+` できない型まで来てしまう。それを防ぐためのもの。

## `Self::Item` は何

`Iterator` トレイトの「関連型 (associated type)」というもの。

```rust
pub trait Iterator {
    type Item;          // ← 「このイテレータが返す要素の型」
    fn next(&mut self) -> Option<Self::Item>;
    // ...
}
```

`v.iter()` の場合、`Item = &i32`（参照を返す）。だから:

- `v.iter().sum::<i32>()` は `Sum<&i32> for i32` を使う（&i32 を集めて i32 を作る）
- `v.into_iter().sum::<i32>()` は `Item = i32` なので `Sum<i32> for i32` を使う

両方とも標準ライブラリで実装済みなので、書く側は何も考えなくて OK。

ジェネリクスとの違い:

- ジェネリクス (`<T>`) = 呼び出し側が型を決める
- 関連型 (`type Item`) = トレイト実装側が型を決める

トレイトを実装した時点で型が固定される ―― だから `Vec<i32>` をイテレータにすると `Item` は `&i32` で固定。

## 記号まとめ

| 記号 | 意味 | どこで指定 |
|---|---|---|
| `<T>` | 型パラメータの宣言 | 関数や型の定義側 |
| `T` | コンパイル時に決まる型 | シグネチャ中で参照 |
| `where T: Trait` | T が満たすべき制約 | 関数や型の定義側 |
| `Self::Item` | 関連型（トレイトが定義する型） | トレイト実装が指定 |
| `::<T>` | ターボフィッシュ（呼び出し側で型を明示） | 呼び出し側 |

## 他言語との対応

| 言語 | 同じ概念 | ターボフィッシュ相当 |
|---|---|---|
| Java | `<T>` ジェネリクス | `Foo.<Integer>method()`（あまり使われない） |
| C++ | `template <typename T>` | `foo<int>()` （Rust の `::<>` と違って `<>` 直書きで OK） |
| C# | `<T>` | `Foo<int>.Method()` |
| Go | `[T any]` (Go 1.18+) | `Foo[int]()`（角括弧） |
| TypeScript | `<T>` | `foo<number>()` |
| Python | `TypeVar` | 型ヒントとして書くのみ |
| Ruby/PHP/古い Go | 概念なし | ― |

Java/C# 経験あれば `<T>` でほぼ同じ。Go 1.18+ のジェネリクスとも近い。Rust は「`<` のあいまいさ回避のために `::<>` を使う」のが少し独自。

## なぜこの設計か

ジェネリクスは「型を抽象化する」ための機能で、Rust 全体に共通する設計目標を支えている:

- 安全性: 「どんな型でも入れていい」ではなく `where` で制約を付ける → 型の安全性を保ったまま抽象化
- ゼロコスト: ジェネリクスは「モノモーフィゼーション」と呼ばれる仕組みで、コンパイル時に型ごとの専用関数を生成する → 実行時の dynamic dispatch なし、性能はベタ書きと同じ
- 書き味: 1 つの関数で複数の型に対応 → API のシグネチャがシンプルに

`v.iter().sum()` がパッと書けるのは、このジェネリクス + トレイト境界 + 型推論の合わせ技。

## 章 05 で深掘りすること

- 自分でジェネリック関数を書く: `fn largest<T: PartialOrd>(list: &[T]) -> &T`
- 自分でトレイトを定義する
- トレイト境界の複数指定 (`T: Display + Clone`)
- ライフタイムパラメータと型パラメータの組み合わせ
- ジェネリクスのコンパイル時展開 (monomorphization)
- トレイトオブジェクト (`dyn Trait`) との対比

今は「`<S>` は型パラメータ、`::<>` はターボフィッシュ」と読めれば OK。

## まとめ

- `<S>` は型パラメータの宣言、呼び出し側が決める型
- `where S: Sum<...>` はトレイト境界（S が満たすべき制約）
- `Self::Item` はトレイトが定義する関連型
- S は型推論で決まる（左辺の型注釈、戻り値、コンテキストから）
- 型推論が効かない時は ターボフィッシュ `::<T>` で明示
- ジェネリクスは安全性・ゼロコスト・書き味の全部を支える Rust の中核機能
- 本格的には章 05 で深掘り
