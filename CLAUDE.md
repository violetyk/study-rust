# CLAUDE.md

このファイルは Claude Code がこのリポジトリで作業するときの project レベル指示。グローバル設定 (`~/.claude/CLAUDE.md`) と併用される。

## このリポジトリの位置付け

PHP / Go / Python / Ruby 等の経験者が Rust を学習するハンズオン。深掘り型の学習スタイルで進めている。

詳しい全体像は [README.md](README.md) を参照。

## 学習スタイル: 深掘り型

ユーザーは「これなんだろう？」と気になる用語・挙動に出会うと、その場で深掘りして理解したい志向を持っている。これは本人が自覚している学習スタイルで、変える意思はない。

- 解説を求められたら出し惜しみせず詳しく書く（短く済ませない）
- 「先に進みましょう」と急かさない
- 本人が「気にしすぎ?」と不安を漏らした時は、深掘りで蓄積した内容を可視化して安心材料にする

## 解説ファイル (explanations/) の作成ルール

### 作成のトリガー

ユーザーから以下のような発言が出たら `docs/rust-handson/explanations/` に Markdown ファイルを作成する。

- 「これも explanations に入れておきましょう」
- 「保存しておきたいですね」
- 「これは記録しておきたい」

### ファイル名

英語の kebab-case。動詞・概念・対比のいずれかを表す:

- `slice.md`
- `binding-and-shadowing.md`
- `function-return-type.md`
- `parse-and-result.md`

### フォーマット

```markdown
# 解説: <タイトル>

[01-basics.md](../01-basics.md) の <セクション名> の補足。

> <キャッチフレーズ。1 文の対比構造や格言形式が望ましい>

## <小見出し 1>

<本文・コード・表>

...

## まとめ

- 箇条書きで要点
- 関連する他の解説 / 章への参照
```

### 推奨される論調と要素

- 「中身は推論、外側は明示」のような対比構造のキャッチコピーを冒頭の引用ボックスに置く
- Rust の設計哲学・方針を 1 〜 2 行添える（ユーザーは設計理由を理解したい志向）
- 他言語（PHP / Go / Python / Ruby / C / C++）との対応表を入れる
- メモリ図は ASCII で、フロー図は Mermaid で（ASCII の方がメモリレイアウトには向く）
- 関連する他の explanation ファイルへのクロスリンクを張る

### 既存解説のキャッチフレーズ一覧

リポジトリのトーンを揃えるための参考。

## 章本文の編集

### explanations へのリンク

各セクションの末尾に `詳しく: [タイトル](explanations/xxx.md)` の形式で追加する。複数ある場合は `｜` 区切り:

```
詳しく: [A](explanations/a.md) ｜ [B](explanations/b.md) ｜ [C](explanations/c.md)
```

### 章本文への加筆

ユーザーから「これ書いておきたい」「これも本文に」と言われた時は本文を直接編集する。explanations に切り出すかどうかは内容のボリューム次第:

- 数行で済む補足 → 章本文に追加
- 1 セクション以上の解説が必要 → explanations に切り出して詳しくリンク

## コードの慣習

### `code/chXX-name/` の構成

各章ごとに `cargo new chXX-<name>` で作成。例: `ch00-hello`、`ch01-basics`。

### 演習コードの構造

`src/main.rs` に章の例（実験コード）と演習を共存させる。演習は関数として分離:

```rust
fn main() {
    // 実験スペース（章の例）...
    
    exercise_1_1();
    exercise_1_2();
    exercise_1_3();
}

fn exercise_1_1() {
    // 演習 1-1 のコード
}

fn exercise_1_2() {
    // 演習 1-2 のコード
}
```

メリット: 各演習が独立、コメントアウトで個別実行可能、後から見返せる。

### clippy 警告の扱い

学習目的で意図的にイディオムから外したコードを書くことがある（`return x;` を見せる、`let mut x = 5; x = x + 1;` を `x += 1` にしないなど）。clippy 警告は学習の記録として残す方針。

- 一律に `#[allow(...)]` で抑制しない
- 警告内容自体が学びになる
- 「Rust 流ではない」を毎回コンパイル時に思い出せる効果がある

ただしユーザーから「これは抑制したい」と言われた時は別。

## 章を閉じる手順

章を終える時の決まった流れ:

1. 章本文のチェックリストを `[x]` で埋める
2. 必要に応じて explanations を追加・更新
3. 2 コミットに分けて作成
   - `docs: 章XX を完了` — 章本文と explanations
   - `feat(code): 章XX 演習A〜B 完了` — `code/chXX-...` の演習コード
4. push して `origin/main` と同期

コミットメッセージは Conventional Commits 風で日本語可。Co-Authored-By を含めるが、グローバル設定があるならそちらに合わせる。

## ユーザーとの対話

### 区切り感を作る

長い深掘りセッションが続いた時、章末の演習・チェックリスト・コミットで「閉じ」を作る提案をすると区切り感が出る。本人も気持ちよく次に進める。

### 概念の繋がりを示す

新しい概念が出てきた時、過去に書いた explanations と繋げる。「これは ch01 でやった『持ち主』の話の延長」のように。これによって学習が点ではなく線になる。
